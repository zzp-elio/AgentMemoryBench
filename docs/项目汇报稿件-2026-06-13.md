## 开场

老师好，我来汇报一下 Agent Memory Benchmark 这个项目的当前进度。

这个项目我们的目标是构建一个**可复现、可扩展、可审计**的 Agent Memory 评测框架。简单说，就是让不同的 benchmark 和不同的记忆系统 method，能通过同一套流程来评测，而不是每次都为新组合复制一整份代码。

目前我们聚焦在 conversation-QA 这一个 task family 上，也就是对话问答场景下的长期记忆评测。Phase 1 以 LoCoMo benchmark 为第一优先，已经接入了 Mem0 OSS 和 MemoryOS 两个 method。

下面我按模块汇报已完成的工作。

---

## 一、核心数据模型与接口设计

首先我们建立了一整套数据实体体系。从大到小是 Dataset → Conversation → Session → Turn，加上 Question 和 GoldAnswerInfo，共 12 个强类型 dataclass。

这里的关键设计是**公有和私有的强制隔离**。Question 是给 method 看的公开问题，GoldAnswerInfo 包含标准答案和 evidence，只能给 evaluator 看，绝不能进 method。我们在数据模型层面就做了隔离——Conversation 有个 to_public_dict 方法，序列化时显式不包含 gold_answers。这比靠程序员自觉要可靠得多。

接口方面定义了三个抽象类。BaseMemorySystem 是每个 method 必须实现的，只有 add 和 get_answer 两个方法，非常精简。BaseResumableMemorySystem 是可选的逐 turn 续写接口，BaseMemoryRetriever 是可选的检索接口。这三个接口的层次划分让 method 接入的成本非常低。

---

## 二、数据安全：四层防护

项目最核心的质量保证是防止标准答案泄漏给 method。我们做了四层防护。

**第一层**在数据模型层：刚才说了，Conversation.to_public_dict() 在序列化时就剥离了私有字段。

**第二层**在 runner 重建层：即使数据模型做了隔离，我们在 runner 里还是用 copy.deepcopy 逐字段重建了公开副本，gold_answers 直接设为空字典。这层防护是防止 Python 对象的动态属性意外泄漏。

**第三层**是黑名单扫描层：一个递归函数叫 validate_no_private_keys，它会扫描任意嵌套的字典和列表，做 case-insensitive 的 key 匹配。预定义了 12 个私有键名，像 answer、gold_answer、evidence、label 这些。如果发现泄漏，不仅抛异常，还给出完整的 JSONPath 定位，方便排错。

**第四层**是 manifest 检查：method 的运行清单里额外禁止 api_key、secret、token、password 这些敏感字段，防止把凭据写进公开产物。

每一层都是独立的安全网，即使某一层失效，后面还有兜底。

---

## 三、异常体系与数据校验

我们建立了项目专用的异常层次。所有异常继承自 MemoryBenchmarkError，CLI 在最外层统一捕获。

有七种专项异常。DatasetNotFoundError 会告诉你缺失了哪个 benchmark 的什么路径。UnknownBenchmarkError 不仅报错，还会列出所有已支持的 benchmark，用户可以立刻知道怎么改命令。ConfigurationError 覆盖了配置缺失、类型错误、未知 key 等情况。DatasetValidationError 覆盖数据校验失败，比如字段缺失、ID 重复、question 和 gold answer 不对齐。DataLeakageError 是私有数据泄漏时抛的。

数据校验方面，写完了一个完整的 validate_dataset 函数，它会递归检查每个 conversation 的必填字段、ID 唯一性、question 和 gold answer 的双向对齐。一个常见的坑是 question 的 conversation_id 和父 conversation 不一致，这个校验能直接拦截。

## 四、日志与可观测性

长实验跑起来要让人知道进度，出了事要知道在哪里出的。我们做了一套完整的可观测性。

RunLogger 同时写三路输出：终端用 Rich 做彩色格式化，即使 markup 语法写错了也不会崩，自动降级为纯文本。同时追加 run.log 文本日志和 events.jsonl 结构化事件。事件类型包括 run_started、conversation_ingested、conversation_failed、question_answered 等，每条带 UTC 时间戳，可以直接用 jq 分析。

ProgressReporter 封装了 Rich 进度条。三阶段展示：Ingest conversations → Answer questions → Completed。每个阶段显示当前进度、总数、正在处理的 conversation ID 和 question ID，还有耗时计时器。同时每 1 秒写一次 progress.json 快照，用的也是原子写入——先写临时文件，fsync 强制落盘，然后 os.replace 原子替换。即使关了终端或者 SSH 断了，后面重新打开这个 JSON 就能知道上次跑到哪了。

还有一个 RunContext 类，是不可变 dataclass，管理标准目录布局。长实验的标准产物目录结构是这样的：outputs 下按 run_id 分子目录，里面有 artifacts、logs、checkpoints、summaries、method_state 五个子目录。RunContext 创建时自动建好，resume 预检时可以关闭这个行为以避免副作用。

## 五、原子存储与断点续跑

存储层面我们做了原子写入和崩溃恢复。

所有 JSON 和 JSONL 的覆盖写入都通过原子操作：先写到同目录的隐藏临时文件，flush 加 fsync 确保落盘，然后 os.replace 做原子替换。这样即使进程在写入中途崩溃，磁盘上也不会留下截断的目标文件。

对于 append-only 的 JSONL，我们专门实现了崩溃恢复逻辑。如果进程在追加一行时崩溃，文件末尾会留下一个没有换行终止符的半行，JSON 解析会失败。read_jsonl 在 recover_torn_tail 模式下，会检查最后一行是不是没写完整，如果是就静默丢弃，然后从这个位置继续跑。中间行的损坏和已经有换行符的尾行损坏仍然会报错，因为我们不能判断那是真损坏还是数据本身有问题。

断点续跑系统分两层。第一层是 conversation 级别的 coarse checkpoint——conversation_status.json 记录每个 conversation 是 completed 还是 failed。第二层是针对支持逐 turn 续写的 method 的 TurnIngestCheckpoint。这是一个三态状态机：ready → in_flight → completed。每次调用 method 前标记 in_flight，返回成功后标记 completed。这里有个重要的安全决策：in_flight 状态绝不自动 resume。因为超时的请求可能在服务端已经被处理了，我们没法判断，所以必须人工确认。

Resume 时的 manifest 校验是最严格的一步。我们会构造当前的新 manifest，包含 dataset 的 SHA-256、完整 policy（多少 worker、多少 question limit）、完整 method 配置（所有参数），和磁盘上已有的 manifest 做逐字段全量比对。完全相等才允许 resume。任何字段变了——哪怕只是 worker 数从 1 变成 2——都拒绝。不猜测、不妥协。

另外，resume 时已完成的 conversation 通过 load_existing_conversation_state 恢复到 method 内存，不会重复 add。而且传入 method factory 的数据通过 _make_public_conversation 重建了公开副本，gold_answers 设为空字典，确保私有数据不会跟着 resume 链路进到 method。

还有一点，我们在创建目录和构造 method 之前做了一个只读 preflight 预检——先做所有校验，通过了才建目录、才读 .env、才构造 method。这样非法输入不会留下任何副作用。

## 六、通用 Prediction Runner

Prediction runner 是一个大约 800 行的通用引擎，与具体的 benchmark 和 method 完全解耦。

它分三个阶段执行。第一阶段 Ingest conversations——把 conversation 历史并发写入 method。第二阶段 Answer questions——按 conversation 并发回答问题。第三阶段 Completed——汇总结果。

并发模型用的是 ThreadPoolExecutor。每个 worker 内部保证 add 完成后才能回答问题。不同 conversation 可以并行。Worker 返回的是不可变的 _ConversationAnswerBatch tuple。协调线程通过 as_completed 收集结果，然后串行提交到共享产物——这样 worker 之间不会产生写竞争。

每个 conversation 完成后立即原子写入 method_predictions.jsonl 和 question_status.jsonl，不用等所有都跑完。这意味着即使中途崩溃，已经完成的那些 prediction 都已经安全落盘了。

异常处理方面，conversation 写入失败会记录到 conversation_status.json 并写结构化事件，然后重新抛出。question 答题失败同理。失败了不影响其他 conversation 的结果。

还有一层返回结果的校验：method 返回的 AnswerResult 的 question_id 和 conversation_id 必须和输入对齐，answer 不能是空字符串，add 的返回值必须包含预期的 conversation_id。这些都是防止 method 实现有 bug 时静默产生错误数据。

---

## 七、通用 Evaluation Runner

Evaluation runner 是纯 artifact-only 的，不调 method、不读 .env、不需要 API key。

它同时读取三类标准 JSONL——public_questions、method_predictions 和 evaluator_private_labels。然后做严格的集合校验：三者的 question_id 集合必须完全一致，不能多一个也不能少一个。然后从这些记录里重建 Question、AnswerResult 和 GoldAnswerInfo 三个实体，逐字段校验类型——比如 metadata 必须是 dict、evidence 必须是字符串列表。

这样做的好处是：同一批 prediction artifacts 可以被多个不同的 evaluator 复算——比如先用 LoCoMo F1 算一轮，再用 LLM judge 算一轮——不需要重新跑 method。预测和评估彻底分离。

## 八、声明式注册表

我们不维护 method × benchmark 的笛卡尔积白名单。而是用声明式的 capability 注册。

Benchmark 注册时声明：我属于 conversation_qa 这个 task family，我需要 CONVERSATION_ADD 和 ANSWER_GENERATION 这两个能力。Method 注册时声明：我支持 conversation_qa，我提供 CONVERSATION_ADD 和 ANSWER_GENERATION。运行时调 validate_compatibility 函数：task family 必须匹配，required capabilities 必须是 provided capabilities 的子集。不匹配就直接报错，列出缺失的能力。

这样做的好处是，新增一个 benchmark 或 method，只要声明好能力，自动就能和现有的 method/benchmark 组合。不需要改任何 registry 代码。

目前 Benchmark registry 注册了 LoCoMo（prediction 已开启）和 LongMemEval（adapter 可用但 prediction 还未开启）。Method registry 注册了 Mem0 和 MemoryOS，各有 14 个冻结字段，包括 task_family、capabilities、system_factory、source_identity_factory、workload_estimator 等。Evaluator registry 注册了 locomo-f1（离线，不需要 API）和 locomo-judge（需要 API，支持 compact 和 detailed 两种 judge profile）。

## 九、配置体系

配置分三层。

最底层是 .env——只放 secret。OPENAI_KEY 和 BASE_URL。load_openai_settings 读取后返回 OpenAISettings 强类型对象，直接 to_client_kwargs 传给 OpenAI SDK，用 to_safe_dict 做脱敏——key 变成 sk-xx 开头 ...xxxx 结尾的格式。

中间层是 TOML profile。我们写了一个通用的 load_typed_profile 函数，大约 300 行，做了 15 项以上的安全校验：TOML 文件存在性、语法检查、顶层只能是 section、禁止 TOML 中声明 profile_name、未知 key 检查、必填字段检查，以及完整的类型校验——支持 str、int、float、bool、Path、list、tuple、set、frozenset、dict、Optional、Union、Any。还支持自动类型转换，比如 TOML 不区分整型和浮点，我们看到 float 注解就把整数转成浮点。

当前有三份 TOML 配置：configs/methods/mem0.toml 包含 Mem0 的 smoke 和 official_full 参数，configs/methods/memoryos.toml 包含 MemoryOS 的全部论文参数——capacity、threshold、top_m_segments、retry 等约 20 个字段，configs/evaluators/llm_judge.toml 包含 compact 和 detailed 的 judge profile。

最上层是 CLI 覆盖。优先级是：代码默认值 < 命名 profile < 用户实验 TOML < CLI 显式参数。

还有一个重要的成本保护机制。所有需要真实 API 的 method 必须传 --confirm-api。official-full profile 因为会调用大量 API，还额外要求 --confirm-full。这个检查是在读取 profile、dataset 和 .env 之前的——也就是说，如果你没有确认付费，我们连配置都不读、目录都不建，零副作用。

MemoryOS 还额外提供了一个 workload_estimator，在 dataset 确定后、method 构造前，估算这次运行会产生多少个 update batch，写入 manifest，用户在付费前就能看到大概的工作量。

## 十、统一命令行入口

CLI 提供三个子命令：predict 只生成回答，evaluate 只读 artifact 算指标，run 是两者的组合。

CLI 的架构是三层分离的。main.py 只做 argparse 解析，不读数据不调 method。commands.py 接收强类型 PredictCommand 或 EvaluateCommand，负责 registry 查询、成本确认，然后调 runner。run_prediction.py 里的 run_registered_conversation_qa_prediction 是核心装配函数，把 benchmark registration、method registration、capability 校验、TOML profile、adapter、dataset、workload estimate、settings、manifest preflight、method factory 和 generic runner 全部串联起来——总共 13 个严格有序的步骤。

所有异常在 CLI 层统一捕获，以红色 Error: 前缀输出到 stderr，exit code 为 2。--debug 开关可以重新抛出异常查看完整 traceback。

还有一个设计细节：CLI 的 --method choices 和 --benchmark choices 是通过 registry 动态获取的。register 一个新 method 后，它自动出现在 CLI 的可选项中，不需要改 main.py。同样 --benchmark 只显示 prediction_enabled=True 的 benchmark。

## 十一、双 Method 集成

我们接入了两个 method。

Mem0 OSS 用的是 vendored 的 2.0.4 版本源码。约 26500 行的 adapter 实现。它同时实现了 BaseMemorySystem 和 BaseResumableMemorySystem。memory namespace 的语义是：一个 conversation_id 对应一个 Mem0 逻辑 namespace，不做双 speaker namespace 的复刻。speaker 信息保留在 content 和 metadata 中。逐 turn 写入时通过 on_turn_started 和 on_turn_completed 回调与 checkpoint store 协作。

MemoryOS 用的是 vendored 的官方 eval 目录源码。约 40000 行的 adapter。实现 BaseMemorySystem 的 add 和 get_answer。MemoryOSPaperConfig 对所有论文参数做了强类型校验——capacity 必须是正整数、threshold 必须在 0 到 1 之间、heat_threshold 必须非负、retry 参数要满足约束等等。最关键的一点是：我们没有修改第三方源码。论文的 retrieval_top_m_segments 参数通过 functools.partial 绑定到官方 search_sessions_by_summary 的 top_k 参数上。对 resume，我们会在加载官方状态文件前预检：short-term 文件必须存在且是合法 JSON，mid 和 long 如果存在也必须是合法的。

另外两个 method 都实现了 capability 和能力声明的一致性检验。曾经有个 review 发现了 MemoryOS 声明了 MEMORY_RETRIEVAL capability 但实际上没有实现公共 retrieve 方法——类似的问题也在 Mem0 上发现，两个都已修复，现在只声明真正实现了的 add 和 answer 能力。

## 十四、当前阻塞与下一步

目前唯一的阻塞项是 Task 7 的最终 review 发现的一个 source identity 问题：MemoryOS 的 source identity 当前只计算了第三方 eval 源码的哈希，没有包含本项目 wrapper adapter。如果只改 wrapper 不改第三方源码，manifest 哈希可能不变，导致错误地允许旧状态 resume。修复很简单——把 wrapper 的哈希纳入组合 identity，加对应的 TDD 测试，reviewer 复审通过就可以正式收口这个 migration 计划。

API 充值后的计划是：先跑 Mem0 official-full，生成正式 prediction；然后用 artifact-only evaluation 离线计算 LoCoMo F1，可选跑 LLM judge；再跑 MemoryOS official-full；最后做双方法的正式 benchmark 对比。

中期计划是开放 LongMemEval 的 prediction、做到两个 benchmark 两个 method 的全闭环，以及一些工程收尾——benchmarks 目录迁移、tests 分类。

远期是效率评测——Retrieval Latency、Memory Context Tokens、Memory Update Cost——以及第二种 task family 的支持，但这些要等真实需求出现再启动。

## 总结

以上就是项目当前的全部进度。总体来说，385 项测试全部通过，编译零错误。我们从零散脚本演进到了一个可扩展的完整评测框架。框架的核心价值在于：新增 benchmark 或 method 只需要实现标准接口并注册，就能自动获得 prediction、resume、evaluation 的完整能力，不需要再为每个组合复制一套专用 runner。谢谢老师。
