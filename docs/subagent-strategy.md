# Subagent 使用策略

本文规定 Codex subagent 与 OpenCode subagent 的选择、升级和验收规则。目标是在保证
工程质量的前提下降低额度消耗，而不是单纯追求最低成本或最高模型规格。

## 派发原则

1. 先判断是否真的需要 subagent。主线程下一步立即依赖的紧耦合阻塞任务，通常由主线程完成。
2. 派发具体、边界清晰、能产生可检查结果的任务，不派发宽泛的“研究整个项目”。
3. 默认串行推进。只有子任务没有前置依赖、写集互不冲突、结果可独立验收时才并行派发；
   不能为了表面速度并行处理实际存在依赖的任务。
4. 选择能可靠完成任务的最低合理模型和 reasoning effort，不默认使用最高规格。
5. 同一任务先小范围试用；根据结果质量、返工次数和 review 发现动态调整后续选择。
6. Subagent 报告完成不等于任务完成。主线程必须检查实际改动并运行必要验证。
7. 实现任务按风险使用 TDD 和 review，不再固定执行 implementer -> spec review ->
   code-quality review 流水线。

## Review 分级

- 低风险机械修改：主线程检查 diff，并运行直接相关测试；通常不派独立 reviewer。
- 普通局部实现：完成 focused tests 后由主线程自检；在完整功能切片结束时统一做一次
  综合 review。
- 高风险契约：涉及 resume、私有数据、指标计算、公共协议或昂贵实验复用时，可在该
  契约完成后单独做一次 reviewer 审查，但不默认拆成 spec 和 quality 两轮。
- 阶段验收：运行 full suite，并做一次最终整体 review。
- Reviewer 发现真实问题后进入 TDD 修复，再交同一 reviewer 复审；不额外增加重复 reviewer。

Review 的目标是发现高价值问题，不是增加流程节点。任务仍须逐步完成和验证，不因减少
review 而跳过未完成步骤或提前关闭阶段。

## Codex 模型选择基线

| 任务类型 | 默认模型 | 默认推理 | 典型任务 |
| --- | --- | --- | --- |
| 短程机械任务、单文件、完整规格 | `gpt-5.4-mini` | `high` | 补注释、改固定路径、增加简单测试 fixture |
| 短程且边界清晰的小型实现 | `gpt-5.4-mini` | `xhigh` | 独立 helper、局部 bugfix、明确的 TDD 任务 |
| 一般跨文件实现或 review | `gpt-5.4` | `medium` | runner/storage 集成、spec review、质量审查 |
| 长程或上下文较大的实现/review | `gpt-5.4` 或 `gpt-5.5` | `medium` | 多文件实现、较长历史、跨模块审查 |
| 复杂调试或重要契约变更 | `gpt-5.4` 或 `gpt-5.5` | `high` | 状态机、resume、一致性、隐私边界 |
| 关键架构、关键契约或最终整体审查 | `gpt-5.5` | `xhigh` | 跨模块架构、发布前最终 review、失败代价很高的判断 |

`gpt-5.4-mini` 只用于短程、小上下文、边界清晰的任务，并且只使用 `high` 或
`xhigh`；禁止把长历史、大量文件或长程连续推理任务交给 mini。长程任务根据难度在
`gpt-5.4 medium/high` 与 `gpt-5.5 medium/high` 之间选择；关键性任务使用
`gpt-5.5 xhigh`。优先顺序仍是先缩小任务和完善规格，再提高模型/推理强度。

这只是起始基线，不是硬编码规则。任务风险、修改范围、失败代价和历史表现优先于表格。

## 动态升级与降级

出现以下任一情况时，下一次同类任务至少升级一个档位：

- 未按明确规格完成，或遗漏关键边界。
- 测试是假绿、没有按 TDD 产生有效 RED。
- review 连续发现同类缺陷。
- 需要主线程大幅重写 subagent 结果。
- subagent 无法解释其改动或验证证据不足。

升级顺序优先为：

```text
缩小任务范围并补齐验收条件 -> 提高 reasoning effort -> 提高模型级别
```

出现以下情况时，可以尝试降级以节省额度：

- 同类任务连续至少两次一次通过 spec/quality review。
- 任务可完全由固定步骤、现有模式和明确测试约束。
- 改动范围小，失败成本低，主线程验证便宜。

每次升级或降级都应根据任务结果，而不是凭感觉长期固定。

## OpenCode 的未来定位

用户明确启用后，OpenCode 优先承担：

- 机械式文档同步、路径核对和格式修正。
- 已有明确测试与实现说明的小范围编码。
- 大量重复性扫描、清单生成和只读核验。
- Codex 额度不足期间，可独立推进且容易验收的任务。

暂不优先交给 OpenCode：

- 新架构设计、模糊需求澄清和关键技术决策。
- 隐私边界、数据泄漏、实验正确性等高风险判断。
- 最终整体 review。
- 需要跨多个复杂模块进行创造性重构的任务。

OpenCode 结果必须视为候选改动：先检查 session messages 和 diff，再由 Codex 或测试验证。
如果多次表现稳定，可逐步扩大任务范围；如果返工率高，则缩小范围或改用 Codex subagent。

## 任务记录

长任务的 handoff 应记录：

- subagent 类型、模型、reasoning effort 和任务范围。
- 完成状态与验证证据。
- review 发现和返工次数。
- 对以后同类任务应升级、保持或降级的结论。

这些记录用于形成项目自己的模型选择经验，避免每个新窗口重新猜测。
