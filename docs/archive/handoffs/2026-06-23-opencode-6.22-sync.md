## 代码变更复核

### efficiency by-question 聚合

根因成立：`QuestionEfficiencyObservation` 到达时会覆盖此前已经按
`LLMCallObservation` 聚合出的 question record，导致 A-Mem 的
`efficiency_by_question.prediction.json` 中 answer/retrieval LLM token 全 0。

当前修复是 merge 已有计数器，不再覆盖。该修复合理。

### `--max-new-conversations` 进度显示

根因成立：work plan 本身只选择了预算内 conversation，但 progress/logger 用 full dataset
规模显示，导致 1-conv run 显示为 500 conversations/questions。

当前修复把 progress/logger total 改成 work-plan-aware total。该修复合理。

### artifact evaluation 支持分批 prediction

根因成立：`public_questions.jsonl` 和 `evaluator_private_labels.jsonl` 保存全量问题，
而分批 prediction 只保存已回答子集。旧 evaluator 要求三套 question id 完全一致，
导致 1-conv cost pilot 不能直接 evaluate。

当前修复改为：

- public ids 必须等于 private ids。
- prediction ids 必须是 public/private 的子集。
- evaluator 只评有 prediction 的 question。

该修复合理，符合分批 resume 实验语义。

## Mem0 `reference_date` 状态

2026-06-23 追加结论：**已修复后续 run 的完整日期传递**。

修复前代码事实：

- Mem0 LoCoMo 写入时会把 `[Session time: ...]` 放进 message content。
- Mem0 extraction 调用时会通过 `_observation_time_prompt(session.session_time)` 给提取器传入
  observation time。
- Mem0 LongMemEval answer prompt 正确传 `question.question_time`。
- 但 Mem0 LoCoMo answer prompt 的全局 `reference_date` 来自：
  `metadata["reference_date"]` -> `metadata["question_reference_date"]` ->
  `_reference_year_from_memories(memories)` -> `"2023"`。
- 修复前 `Conversation.metadata` / `_conversation_metadata` 没有写入完整 `reference_date`，
  因此全局 reference_date 通常只有年份，不是官方 run.py 的完整 reference date。

当前修复：

- `Mem0._conversation_public_metadata()` 优先保留显式 `reference_date` /
  `question_reference_date`，否则使用公开 history 中最后一个有值的 session time。
- LoCoMo 与 LongMemEval 两条写入路径都统一缓存这份公开 metadata。
- `tests/test_mem0_adapter.py::test_get_answer_uses_mem0_locomo_official_answer_prompt`
  已覆盖官方 prompt 中出现
  `These conversations took place around 2023-05-10T10:00:00`。

影响判断：该修复不自动改写现有 `outputs/mem0-locomo-full-v4/`；full-v4 可继续作为历史
baseline，但如果最终论文/报告要求最严谨 Mem0 LoCoMo 官方 prompt 复现，应使用新 run_id
重跑 Mem0 LoCoMo。
