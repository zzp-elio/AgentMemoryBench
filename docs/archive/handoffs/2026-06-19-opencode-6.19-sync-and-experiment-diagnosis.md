## OpenCode 6.19 主要改动

- Rich 进度输出：
  - `CalibrationProgressMonitor` 使用统一 Rich `Live(Table)`。
  - `ProgressReporter` 和 monitor 改为使用 `sys.__stdout__`，降低
    `contextlib.redirect_stdout` 对 Rich 的影响。
- `calibrate-smoke`：
  - 并行模式禁用 child run Rich progress。
  - `--max-parallel-runs` 扩展到 `3`。
- prediction artifact：
  - `conversation_prompts.jsonl` 记录 conversation 级 `system_prompt`。
  - `method_predictions.jsonl` 移除重复 `system_prompt`。
  - MemoryOS adapter 不再把 `user_prompt` 写入 metadata。
- evaluator：
  - `run_artifact_evaluation()` 增加通用 `category_breakdown`。
  - LoCoMo judge prompt 对齐 LightMem 官方风格。
  - compact 模式解析 `CORRECT` / `WRONG`。
- Mem0：
  - LoCoMo reader 输出若包含推理链，adapter 提取最后一个 `ANSWER:` 后的最终答案。

## Codex 已修复的问题

### LongMemEval calibrate-smoke 终端最终 pending

根因不是 child run 失败。三个 child run 的 `progress.json` 和 `summary.json` 均显示
completed：

- `outputs/longmemeval-smoke-0619-1324-mem0-longmemeval-s-cleaned`
- `outputs/longmemeval-smoke-0619-1324-amem-longmemeval-s-cleaned`
- `outputs/longmemeval-smoke-0619-1324-lightmem-longmemeval-s-cleaned`

真实根因：monitor 用 base run id 查找：

```text
longmemeval-smoke-0619-1324-mem0-longmemeval
```

但 registered prediction 实际写入 concrete variant child run：

```text
longmemeval-smoke-0619-1324-mem0-longmemeval-s-cleaned
```

因此 monitor 读不到 `progress.json`，终端表格保持 pending。

Codex 已新增红绿测试并修复：

- `tests/test_calibration_progress_monitor.py::test_monitor_reads_variant_child_progress_by_base_run_id`
- `CalibrationProgressMonitor._resolve_progress_path()` 支持 exact path 不存在时查找
  `base-* / checkpoints / progress.json`。

验证：

```bash
uv run pytest tests/test_calibration_progress_monitor.py tests/test_cost_calibration_smoke.py -q
# 24 passed
```

成功实验继续评测。
