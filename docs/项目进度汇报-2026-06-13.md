# Agent Memory Benchmark 项目进度汇报

日期：2026年6月13日

---

## 一、项目概述

本项目构建了一个**可复现、可扩展、可审计的 Agent Memory Benchmark 评测框架**。框架读取 benchmark 原始数据，转换为统一数据模型，调用 memory method（记忆系统）生成回答，并以标准实验产物形式保存结果。当前实现的 conversation-QA task family 已接入 **LoCoMo** benchmark 和 **Mem0 OSS**、**MemoryOS** 两个 method。

## 二、已完成工作

### 2.1 核心数据模型与接口

定义了完整的 conversation-QA 实体层级（`Dataset → Conversation → Session → Turn`），共 12 个强类型 `@dataclass`。其中 `Question` 是 method 可见输入，`GoldAnswerInfo`（含 gold answer、evidence）是 evaluator 私有数据，两者从数据模型层面强制隔离——`Conversation.to_public_dict()` 显式清除 `gold_answers`。

定义了三个抽象接口：`BaseMemorySystem`（`add` + `get_answer`）、`BaseResumableMemorySystem`（逐 turn 安全续写）和 `BaseMemoryRetriever`（可选检索）。

### 2.2 公有/私有数据隔离机制

多层防护确保私有数据不进入 method：
- **数据模型层**：`Conversation.to_public_dict()` 显式剥离 `gold_answers`
- **Runner 层**：`_make_public_conversation()` 逐字段重建公开副本（`copy.deepcopy`），传入 method 前清空 `gold_answers={}`
- **校验层**：`validate_no_private_keys()` 递归扫描任意嵌套结构，case-insensitive 匹配 12 个私有键名黑名单（`answer`、`evidence`、`gold_answer` 等），发现泄漏时抛 `DataLeakageError` 并给出完整 JSONPath
- **Manifest 层**：method manifest 额外检查 `api_key`、`secret`、`token`、`password` 等敏感键名

### 2.3 领域异常体系（7 种专项异常）

| 异常 | 用途 |
|------|------|
| `MemoryBenchmarkError` | 所有自定义异常的基类 |
| `DatasetNotFoundError` | 原始数据缺失（含 benchmark 名和路径） |
| `UnknownBenchmarkError` | 未注册 benchmark（列出所有已支持项） |
| `AdapterAlreadyRegisteredError` | 名称冲突 |
| `ConfigurationError` | 配置缺失/非法/类型错误/未知 key |
| `DatasetValidationError` | 字段缺失/重复 ID/question-gold 不对齐 |
| `DataLeakageError` | 私有数据泄漏风险（含 JSONPath 定位） |
| `JudgeOutputError` | LLM judge 输出无法解析 |

所有异常承载足够上下文，CLI 统一捕获并以红色 `Error:` 输出到 stderr。

### 2.4 数据校验系统

三层递进校验：
1. **实体级**：`validate_dataset()` → `validate_conversation()`，检查所有必填字段、ID 唯一性、question/gold 双向对齐、conversation_id 一致性
2. **私有数据级**：`validate_no_private_keys()` 递归检查传入 method 的所有数据
3. **Runner 返回级**：校验 `AnswerResult.question_id/conversation_id` 对齐、`add()` 返回值包含期望 conversation_id、answer 文本非空

### 2.5 日志与可观测性系统

**RunLogger**（`utils/run_logger.py`）：
- 终端：Rich 彩色格式化输出（markup 语法错误时自动降级为纯文本，确保长实验不因日志格式中断）
- 文件：同步追加写入 `run.log`（人类可读，带 UTC 时间戳）
- 结构化事件：同步追加 `events.jsonl`（包含 `run_started`、`conversation_ingested`、`conversation_failed`、`question_answered`、`question_batch_failed`、`run_completed` 等事件）

**ProgressReporter**（`observability/progress_reporter.py`）：
- 终端：Rich 进度条，展示阶段名称（"Ingest conversations" / "Answer questions"）、conversation 进度（含当前 ID）、question 进度（含当前 conversation 和 question ID）、耗时计时器
- 文件：`progress.json` 持久化进度快照，节流写入（1 秒间隔），即使关闭终端也继续写
- 原子写入：临时文件 → `os.fsync()` → `os.replace()` 确保崩溃安全
- 可注入自定义时钟用于测试

**RunContext**（`observability/run_context.py`）：不可变 dataclass，管理标准目录布局（logs/artifacts/checkpoints/summaries/method_state），resume 预检时可关闭 `ensure_directories` 避免副作用。

### 2.6 实验产物存储体系

**标准产物结构**（`outputs/<run_id>/`）：
- `manifest.json`：完整运行元数据，resume 时用于全量比对
- `config.redacted.json`：脱敏配置快照（API key 仅保留 `sk-xx...xxxx`）
- `artifacts/dataset_fingerprint.json`：数据集内容 SHA-256 + 源文件指纹（含大小和哈希）
- `artifacts/public_questions.jsonl`：method 可见问题
- `artifacts/method_predictions.jsonl`：method 回答（含 `question_text`，可独立复算）
- `artifacts/evaluator_private_labels.jsonl`：evaluator-only 私有标签
- `artifacts/answer_scores.<metric>.jsonl`：逐题分数明细
- `logs/run.log` + `logs/events.jsonl`
- `checkpoints/conversation_status.json` + `question_status.jsonl` + `progress.json`
- `checkpoints/ingest_turns/<sha256>.json`：逐 turn 断点
- `method_state/`：method 私有状态
- `summaries/summary.json` + `summary.<metric>.json`

**原子写入**（`storage/atomic.py`）：所有 JSON/JSONL 文件的覆盖写入都通过同目录临时文件 + `os.fsync()` + `os.replace()` 实现原子替换。

**崩溃恢复**（`storage/jsonl.py`）：`read_jsonl(recover_torn_tail=True)` 能识别并丢弃 append-only 文件因崩溃留下的无行终止符损坏尾行，中间坏行仍报错。

### 2.7 断点续跑（Resume）系统

**全量 manifest 校验**：resume 时比对现有 manifest 与新构造的 manifest（含 dataset_sha256、完整 policy、完整 method 配置），逐字段完全相等才允许继续。不匹配直接拒绝，不猜测。

**只读 preflight**：在创建目录和构造 method 前预检 manifest 状态，非法输入不留下空目录或无效 method 状态。

**Conversation 级 breakpoint**：每个 conversation 记录 `completed`/`failed` 状态，每次完成/失败后原子更新 `conversation_status.json`。

**逐 turn checkpoint 状态机**：三态 `ready → in_flight → completed`，每个 conversation 独立 SHA-256 哈希文件。`mark_started()` 严格校验状态转换和 turn index 对齐；`mark_turn_completed()` 验证与 `in_flight` 匹配。**`in_flight` 状态绝不自动 resume**（无法判断超时请求是否已被服务端处理）。

**Resume attach**：resume 时已完成的 conversation 通过 `load_existing_conversation_state()` 恢复到 method 内存，不重复 `add()`。传入前通过 `_make_public_conversation()` 重建公开副本。

### 2.8 通用 Prediction Runner

`run_predictions()` 是与 benchmark/method 无关的引擎（约 800 行），分三阶段执行：
1. **Ingest conversations**：并发写入历史，支持逐 turn checkpoint
2. **Answer questions**：按 conversation 并发答题
3. **Completed**：汇总

并发模型：`ThreadPoolExecutor` 控制并发数，worker 内部保证 `add → questions` 顺序，返回不可变 batch；协调线程用 `as_completed()` 串行提交共享产物，避免竞争。每个 batch 完成后立即原子写入，支持中途崩溃恢复。

prediction 返回结果严格校验（question_id/conversation_id 对齐、answer 非空、add 返回值包含期望 conversation_id）。

### 2.9 通用 Evaluation Runner（artifact-only）

`run_artifact_evaluation()` 纯读取 artifacts 计算指标，不调用 method、不读取 `.env`：
- 从 manifest 校验 benchmark 身份
- 同时读取三类 JSONL（public questions / predictions / private labels）
- 严格校验 question_id 集合完全一致
- 显示重建实体并逐字段类型校验
- 原子写入逐题 score 和 metric summary

### 2.10 声明式能力注册表

不维护 method × benchmark 笛卡尔积白名单，改用声明式契约：

**Benchmark** 声明：`task_family` + `required_capabilities`（如 `CONVERSATION_ADD`、`ANSWER_GENERATION`）+ `prediction_enabled`
**Method** 声明：`task_families` + `provided_capabilities` + `system_factory` + `source_identity_factory` + `workload_estimator` 等 14 个冻结字段
**Evaluator** 声明：`supported_benchmarks` + `requires_api` + `profile_names` + `factory`

`validate_compatibility()` 在运行前校验：task family 匹配 + required ⊆ provided。不匹配 → `ConfigurationError` 并列出缺失项。新增 method/benchmark 自动兼容，无需改 registry 代码。

当前注册：LoCoMo（prediction enabled）、LongMemEval（adapter 可用但 prediction 未开启）；Mem0（allow_smoke_worker_override）、MemoryOS（有 workload_estimator，不可覆盖 worker）；locomo-f1（离线）、locomo-judge（需 API）。

### 2.11 分层配置体系

**Secret 层**：`.env` → `OPENAI_KEY`/`BASE_URL`，`OpenAISettings.to_client_kwargs()` 直接传 OpenAI SDK，`to_safe_dict()` 自动脱敏。`load_path_settings()` 可独立于 API key 读取（离线 evaluator 不需要）。

**TOML Profile 层**：`load_typed_profile()` 约 300 行通用解析器，支持 15+ 项安全校验——文件存在性、TOML 语法、顶层只能含 section、禁止 TOML 中声明 `profile_name`、未知 key 检查、必填字段检查、类型校验（str/int/float/bool/Path/list/tuple/set/frozenset/dict/Optional/Union/Any）、自动类型转换。错误信息可读。

**优先级**：代码安全默认值 < 命名 profile < 用户 TOML < CLI 覆盖。Profile 分 smoke / official-full / custom 三级。

**配置文件**：`configs/methods/{mem0,memoryos}.toml` + `configs/evaluators/llm_judge.toml`。

### 2.12 统一 CLI 入口

```bash
memory-benchmark predict   # 只生成回答
memory-benchmark evaluate  # 只读 artifacts 计算指标
memory-benchmark run       # 组合
```

**成本保护**：在读取 profile/dataset/secret 之前校验 `--confirm-api`（所有 API method 必需）和 `--confirm-full`（official-full 额外要求）。未确认前零副作用（不创建目录、不读 `.env`、不构造 method）。

**Method 选择动态化**：`--method` choices 通过 `list_methods()` 动态获取；`--benchmark` choices 使用 `list_prediction_benchmarks()` 只显示已开放项。新增 method 自动出现在 CLI 中。所有 `MemoryBenchmarkError` 统一捕获为 stderr 结构化错误，`--debug` 可显示完整 traceback。

### 2.13 Unified Prediction Service

`run_registered_conversation_qa_prediction()` 是核心装配函数，13 个严格步骤串联：读取 registration → 拒绝未开放 benchmark → capability 校验 → 成本确认 → TOML profile → adapter → dataset → smoke scope → workload estimate → settings → method manifest → preflight → factory → generic runner。旧 `run_mem0_locomo_prediction()` 保留为兼容转发。

## 三、当前架构

```text
CLI (predict / evaluate / run)
  → Command Service (PredictCommand / EvaluateCommand)
  → Unified Prediction Service
      ├── Benchmark Registry (task family + required capabilities)
      ├── Method Registry (system factory + source identity + capabilities)
      └── Config (.env → TOML → 强类型 dataclass)
  → BenchmarkAdapter.load() → Dataset
  → run_predictions() (manifest preflight → ingest → answer → artifacts)
  → run_artifact_evaluation() (artifacts → evaluator → scores)
  → Immutable Run Directory (manifest + artifacts + checkpoints + logs + summaries)
```
