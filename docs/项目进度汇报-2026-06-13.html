核心数据模型 & 接口

■

12 个核心实体

Dataset → Conversation → Session → Turn，加 Question、GoldAnswerInfo、AnswerResult 等，全部为强类型 @dataclass，支持 JSON 序列化。

@dataclassto_dict()to_public_dict()
■

公有/私有隔离

Conversation.to_public_dict() 显式剥离 gold_answers。Runner 层重建公开副本，清空私有字段。

gold_answers={}copy.deepcopy
■

3 层抽象接口

BaseMemorySystem（add + get_answer）、BaseResumableMemorySystem（逐 turn 续写）、BaseMemoryRetriever（可选检索）。

ABCProtocol@abstractmethod

数据安全：四层防护

#层级机制说明
L1 数据模型层 Conversation.to_public_dict() 显式不序列化 gold_answers，从结构层面杜绝泄漏
L2 Runner 重建层 _make_public_conversation() 使用 copy.deepcopy 逐字段重建公开副本，gold_answers={}
L3 黑名单扫描层 validate_no_private_keys() 递归扫描任意嵌套结构，case-insensitive 匹配 12 个私有键名，给出 JSONPath 定位
L4 Manifest 检查层 _validate_public_manifest() 额外禁止 api_key、secret、token、password 等敏感键名

日志与可观测性

RunLogger

终端 Rich 彩色输出 + run.log（人工可读） + events.jsonl（结构化事件流）。Rich markup 错误时自动降级为纯文本，长实验不中断。

run_started · conversation_ingested · question_answered · run_completed

ProgressReporter

三阶段 Rich 进度条：阶段名称 + conversation 进度（含当前 ID）+ question 进度 + 耗时计时器。同步写 progress.json 快照（节流写入），关闭终端也不丢。

原子写入：tmp → fsync → os.replace

RunContext

不可变 @dataclass(frozen=True)，管理标准目录布局。resume 预检时可关闭 ensure_directories 避免副作用。

logs/ · artifacts/ · checkpoints/ · summaries/ · method_state/

原子存储 + 崩溃恢复

所有 JSON/JSONL 覆盖写入通过同目录临时文件 + fsync + os.replace 原子替换。read_jsonl(recover_torn_tail=True) 自动丢弃崩溃损坏尾行。

tmp → flush → fsync → replace

断点续跑（Resume）系统

全量 Manifest 比对

Resume 时比对现有 manifest 与新建 manifest（含 dataset_sha256、完整 policy、完整 method 配置），逐字段完全相等才允许继续。

schema_version · runner · dataset_sha256 · policy · method

只读 Preflight 预检

创建目录和 method 前预检 manifest 状态。非法输入不留下空目录或无效 method 状态。预检通过后才创建目录、构造 method。

逐 Turn 三态状态机

ready → in_flight → completed。每次转换严格校验状态匹配和 turn index 对齐。in_flight 绝不自动 resume。

TurnIngestCheckpoint · TurnIngestCheckpointStore

Resume Attach

完成的 conversation 通过 load_existing_conversation_state() 恢复到 method 内存，不重复 add。传入前通过 _make_public_conversation() 重建公开副本。

通用 Runner 引擎

Prediction Runner (~800 行)

与 benchmark/method 无关的通用引擎。ThreadPoolExecutor 控制并发，worker 内 add → questions 有序，协调线程用 as_completed() 收集 batch 并串行提交产物。

3-stage pipeline · 原子 batch 提交 · 中途崩溃可恢复

Evaluation Runner (artifact-only)

纯读取 3 类标准 JSONL，从 artifact 记录显式重建实体，逐字段类型校验。不调 method、不读 .env。同一批 prediction 可复算多个 metric。

public_questions · method_predictions · evaluator_private_labels

返回结果多层校验

question_id/conversation_id 对齐校验、answer 非空校验、add() 返回值必须包含期望 conversation_id、method manifest 敏感字段检查。

_validate_prediction() · _validate_public_manifest()

并发隔离策略

Worker 不直接并发写共享 JSONL。Conversation 间并行（ThreadPoolExecutor），conversation 内串行（add → questions → batch return）。

max_workers · as_completed · _ConversationAnswerBatch

分层配置体系

Secret 层

.env → OPENAI_KEY / BASE_URL。OpenAISettings.to_client_kwargs() 直传 SDK，to_safe_dict() 自动脱敏（sk-xx...xxxx）。

python-dotenv · load_dotenv

TOML Profile 层

load_typed_profile() 通用解析器：15+ 项安全校验（语法、未知 key、必填字段、类型检查 str/int/float/bool/Path/list/tuple/set/frozenset/dict/Optional/Union、自动类型转换）。

tomllib · get_type_hints · __post_init__

CLI 覆盖层

优先级：代码默认值 < 命名 profile < 用户实验 TOML < CLI 显式覆盖。Profile 分 smoke / official-full / custom 三级。

configs/methods/ · configs/evaluators/

成本保护与零副作用

付费确认（--confirm-api / --confirm-full）在读取 profile、dataset、secret 和构造 method 之前完成。未确认 = 零副作用。

_confirm_prediction_cost · preflight

统一装配入口

核心函数 run_registered_conversation_qa_prediction()，13 个严格步骤串联：

1. registration 读 2. prediction_enabled 3. compatibility 4. 成本确认 5. TOML profile 6. adapter 7. dataset 8. smoke scope 9. workload 10. settings 11. manifest 12. preflight 13. factory → runner

总结

385 项测试全部通过，编译零错误。已完成从零散脚本到完整可扩展评测框架的工程化转型。
