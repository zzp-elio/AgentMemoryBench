# 工程深度分析：基本功能之外做了什么

> 核心流程确实只有三步：adapter 转换数据 → predict 跑实验 → evaluate 算指标。
> 但我们的代码总量约 **22,000 行**（源码 ~12,000 + 测试 ~10,000）。
> 下面按模块逐一解释：除了跑通基本功能，我们还做了哪些事情，以及为什么需要这些代码。

---

## 总览

| 类别 | 做什么 | 为什么不是"基本功能" |
|------|--------|---------------------|
| 数据安全 | 四层私有数据防护 | 基本功能可以只在注释里写"别传 gold answer" |
| Resume 系统 | 状态机 + 全量 manifest 比对 | 基本功能可以只重跑，不做断点续跑 |
| 崩溃恢复 | 原子写入 + torn tail 检测 | 基本功能可以 `open(f,'w').write(json)` |
| 可观测性 | Rich 进度条 + 事件流 + 快照 | 基本功能可以只 print |
| 配置校验 | 15+ 项类型/范围/必填检查 | 基本功能可以用 `**tomllib.load()` 直接解包 |
| Method 适配 | 安全导入 + retry + 参数注入 + 状态校验 | 基本功能可以写死路径和参数 |
| 成本保护 | 付费前零副作用 preflight | 基本功能可以先跑再报账单 |
| 并发安全 | Worker 隔离 + 协调层串行提交 | 基本功能可以单线程 |
| Registry | 声明式能力兼容 | 基本功能可以用 if/elif 白名单 |
| 边界兼容 | Legacy 共存 + alias 对账 | 基本功能可以直接删旧代码 |

---

## 一、数据安全：四层防护（~400 行）

基本功能只需要保证"adapter 不把 gold_answer 放进 Question"。我们做了四层：

**L1 数据模型层**：`Conversation.to_public_dict()` 显式不序列化 `gold_answers`。这是在数据结构层面封死的，不是约定。

**L2 Runner 重建层**：`_make_public_conversation()` / `_make_public_question()` 等 5 个函数，用 `copy.deepcopy` 逐字段重建新对象。即使调用方在原始 Conversation 上动态挂了一个 `._secret` 属性，deepcopy 也不会带到副本上。传入 method 前 `gold_answers={}` 写死。

**L3 黑名单扫描层**：`validate_no_private_keys()` 是一个递归深度遍历函数，扫描所有嵌套的 dict 和 list。预定义了 12 个私有 key 名（`answer`、`gold_answer`、`evidence`、`judge_label`、`label` 等），做 case-insensitive 匹配。发现匹配就抛 `DataLeakageError`，异常消息包含完整 JSONPath（如 `$.sessions[0].turns[3].metadata.evidence`），能直接定位问题来源。这个检查在每次传入 method 前执行。

**L4 Manifest 检查层**：`_validate_public_manifest()` 对 method manifest 做额外检查——禁止 key 名包含 `api_key`、`secret`、`token`、`password` 等敏感片段。递归检查所有嵌套层级。

**为什么需要四层**：每一层是不可替代的独立安全网。L1 防序列化泄漏，L2 防 Python 对象动态属性，L3 防 metadata 中意外携带私有 key，L4 防 secret 进入公开 manifest。四层同时失效的概率可以忽略。

**另一个细节**：resume 时已完成的 conversation 通过 `_make_public_conversation()` 重建公开副本后才传给 method factory。这是因为 resume 链路需要把已完成 conversation 的状态恢复到 method 内存中（`load_existing_conversation_state()`），如果不先清洗数据，gold answer 可能顺着 resume 链路进入 method 状态。


## 二、Resume（断点续跑）系统（~400 行）

基本功能：崩溃了重跑就行。但我们做了完整的断点续跑。

**全量 manifest 逐字段比对**：resume 时构造一个新的 manifest（包含 `schema_version`、`runner` 标识、`dataset_sha256`、完整 `policy`（max_workers、conversation_ids、question_limit）、完整 `method` 配置），与磁盘上已有的 manifest 做 `.keys()` 和逐字段 `==` 比对。完全相等才允许 resume。任何不匹配——哪怕只是 worker 数从 1 改成 2——都立即拒绝，附带 `"Resume manifest mismatch: dataset, method or run policy changed"` 错误。不猜测、不妥协。

**只读 preflight 预检**：在创建任何目录、写入任何文件、构造 method 之前，先做所有校验。校验通过后才建目录和工厂。非法输入不留下空目录或无效 method 状态。这个顺序是经过 review 发现缺陷后修复的——初版在建目录后才校验，导致错误输入会留下垃圾目录。

**逐 turn 三态状态机**：`TurnIngestCheckpoint` 实现了 `ready → in_flight → completed` 三态转换。`TurnIngestCheckpointStore` 提供了 4 个方法（`mark_started` / `mark_turn_completed` / `mark_conversation_completed` / `load`），每个方法都做了严格的状态转换校验：
- `mark_started` 要求当前状态为 `ready`，且 turn_index 必须等于 next_turn_index
- `mark_turn_completed` 要求当前状态为 `in_flight`，且 current_turn_index/turn_id 必须匹配
- `mark_conversation_completed` 要求所有 turn 已确认，即 `next_turn_index == total_turns`
- `in_flight` 状态绝不自动 resume（无法判断超时请求是否已被服务端处理）

**路径安全**：conversation_id → SHA-256 哈希 → 文件名，防止特殊字符导致的路径逃逸或文件系统兼容问题。`load_completed_conversation_ids()` 同时检查 coarse（`conversation_status.json`）和 fine（逐 turn）两层 checkpoint。

**两层 checkpoint 协调**：conversation 级 `conversation_status.json` 记录 completed/failed；turn 级 `ingest_turns/<sha256>.json` 记录逐 turn 状态。preflight 阶段先读取所有 checkpoint，对 `completed` 的 fine checkpoint 自动修复 coarse 状态，对 `in_flight` 发现不一致直接拒绝。


## 三、原子写入与崩溃恢复（~300 行）

基本功能：`json.dump()` 直接写文件。我们做了完整的原子性和崩溃恢复。

**原子写入**：`atomic_write_json()` 和 `atomic_write_jsonl()` 实现了标准的"临时文件 → fsync → 原子替换"模式。关键步骤是：
1. 在同目录创建 `.filename.tmp` 临时文件
2. 写入完整内容
3. `file.flush()` + `os.fsync(fileno)` 强制内核将数据刷到磁盘
4. `os.replace()` 原子替换（POSIX 保证 `rename` 的原子性）
5. 失败时 finally 清理临时文件

如果进程在步骤 2 中途崩溃：磁盘上只有不完整的临时文件，目标文件完好。如果在步骤 3-4 之间崩溃：最坏情况是留下一个临时文件（下次会被覆盖）。目标文件永远不会是截断的。

**JSONL 崩溃恢复**：`read_jsonl()` 支持 `recover_torn_tail=True` 模式。append-only 的 JSONL 如果进程在写完一行但还没写换行符时崩溃，文件末尾会有一个"无行终止符的损坏尾行"。这个函数会：
1. 读取所有行（保留换行符信息）
2. 定位最后一个非空行
3. 如果该行 JSON 解析失败且没有换行终止符 → 静默丢弃
4. 中间任意行的损坏或已带换行符的尾行损坏 → 仍然报错

这个机制保证了长实验即使中途崩溃，resume 时能够安全地从最后一个完好记录继续。


## 四、可观测性系统（~500 行）

基本功能：`print("running...")`。我们做了三层可观测性。

**RunLogger**：同时输出到终端（Rich 彩色）、文本日志（`run.log`，带 UTC 时间戳，人类可读）、结构化事件流（`events.jsonl`，带 event 名称和 JSON payload）。关键细节：Rich 的 markup 语法如果写错了会抛异常——我们在 `info()` 方法里捕获了 `MarkupError` 并自动降级为纯文本输出，确保长实验不会因为一条日志格式问题而中断。

**ProgressReporter**：Rich 进度条的三阶段展示（"Ingest conversations" → "Answer questions" → "Completed"）。每个阶段实时显示当前 conversation_id、question_id 和耗时计时器。同时每 1 秒（可配置）写入一次 `progress.json` 快照——使用和原子存储相同的"临时文件 → fsync → os.replace"流程。即使关闭了 Rich 终端展示（`enabled=False`），快照仍然会写入。可注入自定义时钟用于确定性测试。

**RunContext + ExperimentPaths**：两个不可变 dataclass 管理标准目录布局。`RunContext` 负责创建目录和记录运行身份（benchmark、method、model）；`ExperimentPaths` 负责派生稳定文件路径。重要细节：`RunContext.create()` 有一个 `ensure_directories` 开关——resume 预检时可以设为 False，避免在 manifest 预检通过前创建目录。


## 五、MemoryOS Adapter 的深度适配（1,077 行）

基本功能：import 官方 eval 模块，调它的 `add` 和 `answer`。实际要处理的问题远多于此。

**安全模块导入**：MemoryOS 官方 `eval/utils.py` 在 import 阶段就执行了 `OpenAI(api_key="")`，新版 OpenAI SDK 会立即报错阻止 import。我们的解决方案：
1. 临时 monkeypatch `openai.OpenAI` 构造函数——空 API key 时注入占位值
2. 导入 7 个 eval 模块
3. 在 `finally` 块中恢复真实 OpenAI 类
4. 清除 `sys.modules` 中导入的 eval 模块名
5. 移除临时插入的 `sys.path`
6. 保存/恢复 `sys.modules` 状态——防止多个 MemoryOS 实例共享被 monkeypatch 的模块

**论文参数注入（不修改第三方）**：通过 `functools.partial` 把 `retrieval_top_m_segments` 绑定到 `search_sessions_by_summary(top_k=...)`、通过直接赋值注入 `heat_threshold`、替换 `compute_segment_heat` 函数、注入 `gpt_client` 和自定义 embedding 函数。所有这些操作都在本项目侧的 wrapper 中完成，不修改 `third_party/` 中的任何文件。

**带退避的重试逻辑**：`_chat_completion_with_retry()` 捕获 `APIConnectionError` 和 `APITimeoutError`，按指数退避重试（最多 8 次，首次等 5s，最大 60s）。回调参数通过 TOML 配置，不是硬编码。

**状态文件校验**：resume 时对 MemoryOS 官方 JSON 状态文件做预检。short-term 文件必需；mid-term 和 long-term 如果存在，必须是合法 UTF-8 JSON 且符合官方顶层 schema（dict 类型）。任何读取/解码/JSON 错误统一包装为 `ConfigurationError`。

**config `__post_init__` 校验**：15+ 个独立校验规则——model 名非空、capacity 必须为正整数（拒绝 bool，因为 Python 中 `bool` 是 `int` 子类）、threshold 必须在 [0,1]、heat_threshold 必须非负、timeout 必须为正、retry_wait 必须为非负、backoff_multiplier 必须 ≥1、max_wait 必须为正、suppress_official_stdout 必须为 bool。这些校验每个都是一个小 helper 函数（`_require_positive_int` 等），在构造时执行，防止无效配置进入长实验。

**确定性 source identity**：`build_memoryos_source_identity()` 对 vendored 官方 eval 源码和本项目 wrapper 做组合哈希。计算方式是逐文件 hash（路径长度 + 路径字节 + 内容长度 + 内容字节），然后对 vendored hash + wrapper path + wrapper hash 做二次组合哈希。输出包含 `source_sha256`（64 位 hex）、`vendored_source_sha256`、`wrapper_sha256`、`file_count`、`files`（按路径排序的完整列表）、`source_mode` 标识和 `wrapper_path`（逻辑路径，非绝对路径）。

**stdout 抑制**：MemoryOS 官方脚本会大量 print 中间结果。`_official_stdout_context()` 用 `contextlib.redirect_stdout` 在需要时把这些输出重定向到 `io.StringIO` 然后丢弃，让 Rich 终端展示保持干净。`suppress_official_stdout=True` 也可以关闭（用于 debug）。

**工作量估算**：`estimate_add_workload()` 静态估算一个 conversation 会产生多少 page 和多少 update batch，不需要实例化 MemoryOS。这个估算结果在付费确认前写入 manifest，让用户在花钱前就知道大概的 API 调用量。


## 六、Prediction Runner 的工程化（826 行）

基本功能：for 循环调 `add` 和 `get_answer`。实际远不止。

**3 阶段流水线**：Ingest conversations → Answer questions → Completed。每个阶段有独立的 Rich 进度条和进度快照更新。

**并发模型**：`ThreadPoolExecutor` + `as_completed()`。Worker 内部保证 `add → questions` 顺序。Worker 只返回不可变 `_ConversationAnswerBatch` tuple。协调线程用 `as_completed()` 收集，**串行**提交到共享产物（JSONL + checkpoint）。这样最大化了 conversation 间的并行度，同时保证了产物写入的原子性和无竞争。

**batch 级原子提交**：每个 conversation batch 返回后立即原子覆写 `method_predictions.jsonl` 和 `question_status.jsonl`（把已知完成的记录全量写入）。不是等所有 conversation 跑完才写——中途崩溃时，已完成的 batch 已经落盘。

**多次私有数据检查**：传入 method 的 Conversation 先经过 `_make_public_conversation()` 重建 → `validate_no_private_keys()` 扫描。传入 method 的 Question 先经过 `_make_public_question()` → `validate_no_private_keys()`。method 返回的 AnswerResult.metadata → `validate_no_private_keys()`。method manifest → `_validate_public_manifest()`（检查 secret key 名）。总共 4 次独立的检查点。

**Prediction 结果校验**：`_validate_prediction()` 校验 `question_id` 对齐、`conversation_id` 对齐、answer 文本非空。`add()` 返回值必须包含预期的 `conversation_id`。

**Conversation 白名单和 question limit**：`PredictionRunPolicy` 支持按 conversation_ids 白名单选择（且会拒绝不存在的 ID），以及 `question_limit_per_conversation` 限制每 conversation 的问题数。

**Resume 状态管理**：从 `method_predictions.jsonl`（带 torn tail 恢复）读取已有预测，从 `conversation_status.json` 读取 coarse 状态，从 `question_status.jsonl` 读取 fine 状态。只对未完成的 conversation/question 继续执行。

**Preflight 预检**：在 `run_predictions` 入口处，先做 `_preflight_prediction_run()` ——校验 dataset + 构造 manifest + 比对已有 manifest。通过后才创建目录和构造 method。非法输入不留下任何副作用。

**Conversation turn count 校验**：`_conversation_turn_count()` 拒绝空 history（0 turn 的 conversation），避免逐 turn checkpoint 出现 `total_turns=0` 的无效状态。

**Root legacy alias 清理**：新 generic run 不创建根目录的 `predictions.jsonl`、`scores.jsonl`、`summary.json`、`conversation_status.json` 等旧版 alias。这些只由 legacy runner 维护，避免新旧路径混乱。


## 七、Evaluation Runner 的严格性（325 行）

基本功能：读 JSONL → 调 evaluator → 输出分数。实际多了很多校验。

**三 artifact 交叉校验**：同时读取 `public_questions.jsonl`、`method_predictions.jsonl`、`evaluator_private_labels.jsonl`。校验三者的 question_id **集合完全相等**。多一个少一个都报错。

**实体重建加类型校验**：从 JSON 记录重建 `Question`、`AnswerResult`、`GoldAnswerInfo` 时，逐字段检查类型——`metadata` 必须是 dict（None -> {}），`evidence` 必须是 list[str]（None -> []），`question_id`/`answer` 必须是 non-empty string，`prediction answer` 空字符串会报专门错误。所有错误统一包装为 `ConfigurationError`。

**Manifest benchmark 校验**：评估前从 manifest 读取 `benchmark_name`，与命令行参数 `expected_benchmark` 比对。防止在错误的 benchmark 上计算 metric。

**重复 question_id 检测**：`_index_records()` 在按 question_id 建索引时发现重复立即报错（包含文件名），而不是静默覆盖或取最后一个。

**并发执行安全**：evaluator 构造是通过 registry factory 每次新建实例，不缓存。不同 metric 的顺序执行不共享状态。


## 八、配置体系的防御性（~650 行）

基本功能：`tomllib.load("config.toml")["smoke"]` → 直接当 dict 用。我们写了一个完整的类型安全解析器。

**`load_typed_profile()` 的 15+ 项校验**：
1. TOML 文件存在性检查
2. TOML 语法合法性（包装 `tomllib.TOMLDecodeError`）
3. 顶层只接受 section/table（拒绝标量和列表）
4. Section 存在性检查
5. 禁止 TOML 中声明 `profile_name`（自动注入，防止配置漂移）
6. 未知 key 检查（报告所有未知 key 名，而不是静默忽略）
7. 必填字段检查（区分"字段缺失"和"字段有默认值"）
8-9. 类型校验支持：`str` / `bool` / `int`（拒绝 bool 子类） / `float` / `Path` / `Any`
10-14. 泛型类型支持：`list[X]` / `tuple[X,...]` / `set[X]` / `frozenset[X]` / `dict[K,V]`
15. Union/Optional 支持
16. 自动类型转换：int → float（TOML 不区分整型和浮点）、str → Path
17. 类型不匹配时生成可读错误信息（`expected int, got str`）
18. dataclass `__post_init__()` 异常透传

**`PathSettings` 的路径安全**：`resolve_third_party_method_path()` 做路径逃逸检查——传入的 method 目录必须是单个名称、不能是绝对路径、不能包含 `..`。解析后的目标路径必须在 method 根目录内。

**Secret 脱敏**：`OpenAISettings.to_safe_dict()` 把 API key 脱敏为 `sk-xx...xxxx`（前 4 位 + 后 4 位）。

**项目根检测**：`_resolve_project_root()` 从当前目录向上遍历，寻找同时包含 `pyproject.toml` 和源码目录的路径。兼容 `src/memory_benchmark/` 和 `memory_benchmark/` 两种布局。


## 九、CLI 层的防御性编程（~800 行）

基本功能：`argparse` 三个子命令。实际做了很多。

**Method/benchmark choices 动态化**：`--method` choices 通过 `list_methods()` 从 registry 动态获取。新增 method 自动出现在 CLI。`--benchmark` choices 用 `list_prediction_benchmarks()` 只显示 prediction 已开启的 benchmark。

**成本保护前置**：`_confirm_prediction_cost()` 在读取 profile、dataset、settings 和构造 method **之前**调用。需要 API → 检查 `--confirm-api`；official-full → 额外检查 `--confirm-full`。确认通过前零副作用。

**Smoke worker override 语义**：MemoryOS 不允许 `--smoke-max-workers` 覆盖（`allow_smoke_worker_override=False` → 显式 `ConfigurationError`："MemoryOS does not support --smoke-max-workers override"）。Mem0 允许覆盖但限制只能用于 smoke profile 和最多 2 个 worker。官方全量 profile 不受 smoke 参数影响。

**Worker 覆盖的安全范围**：`_resolve_smoke_max_workers()` 校验了 4 个条件——`None`（OK，用配置默认值）、非 smoke profile（拒绝）、method 不允许覆盖（拒绝）、超出范围（拒绝）。

**路径逃逸防护**：`_resolve_run_dir()` 校验 run_id 解析后的路径必须在 `outputs/` 内。

**Error boundary + debug toggle**：所有 `MemoryBenchmarkError` 在 `main()` 层捕获，以红色 `Error:` 输出到 stderr。`--debug` 开关可以重新抛出原始异常查看 traceback。区分了错误输出（stderr）和正常输出（stdout）。

**run 命令的 predict → evaluate 串联**：`run` 命令先调 `execute_predict` 获取真实 run_id，再用这个 run_id 调 `execute_evaluate`。防止了用户手动传的 run_id 与实际产生的 run_id 不一致。


## 十、并发安全设计

**Worker 职责边界**：Worker 只能调 method（add + get_answer），返回不可变 batch。Worker 不能写文件、不能访问共享状态、不能读 `.env`。

**协调层单线程提交**：`as_completed()` 返回结果的顺序是不确定的，但协调线程是单线程处理每个结果的——读完一个 batch → 原子覆写 JSONL → 原子覆写 checkpoint → 更新进度条。不存在两个 worker 同时写的情况。

**Conversation 内串行保证**：`_answer_conversation_questions()` 在 for 循环中串行调 `get_answer()`，确保同一 conversation 的问题按顺序回答。

**Conversation 间隔离**：每个 conversation 有独立的内存状态（MemoryOS 的 `_states` dict、Mem0 的 `run_id=conversation_id`）。不同 worker 不会修改彼此的 memory namespace。


## 十一、Legacy 兼容与新旧边界

**旧 runner 保留但降级**：`memoryos_locomo_full.py`（~1,563 行）和 `memoryos_locomo_smoke.py`（~246 行）保留，docstring 和顶部注释明确标注"只用于复查历史 run，新实验请用统一入口"。保留了 legacy alias（`predictions.jsonl`、`scores.jsonl`、`summary.json`、`conversation_status.json`），但新 generic run 不创建这些 alias。

**新旧不能混跑**：legacy runner 的 checkpoint/resume 逻辑与 generic runner 不同（旧 runner 有专用的 `_verify_run_identity`、`_resolve_existing_state` 等函数）。新代码不引用旧 runner 的 resume 逻辑，旧 runner 不引用新 registry。

**artifact 可复用**：新 generic run 写的 `method_predictions.jsonl` 包含了 `question_text` 字段，可以直接被 `run_artifact_evaluation()` 离线复算 F1 和 LLM judge，不需要依赖新/旧 runner 的身份信息。


## 十二、测试工程（10,441 行 / 30 个文件）

**Fake/mock 系统**：为了在不调用真实 API 的情况下测试 prediction runner、resume、evaluation 等逻辑，我们写了 fake MemoryOS、fake Mem0、fake adapter 和临时 run 目录的测试辅助。这些 fake 需要模拟真实行为（包括 add 的副作用、get_answer 的返回值）。

**TDD 流程**：每个 task 都是 RED → GREEN → REFACTOR 的标准 TDD。先写失败的测试（验证错误类型和错误信息），再实现功能，最后重构。测试不仅验证功能，还验证错误消息的可读性和类型。

**Pytest markers 和 API 隔离**：`unit`、`integration`、`memoryos`、`api`、`slow`、`expensive` 六个 markers。默认 `uv run pytest` 等于 `uv run pytest -m "not api"`，确保零 API 调用。

**文档规范测试**：`test_documentation_standards.py` 自动检查所有 Python 文件的中文 docstring 是否符合规范。


## 十三、工程基础设施

**src-layout**：第一方代码迁入 `src/memory_benchmark/`，第三方源码迁入 `third_party/methods/`。根目录 `pyproject.toml` 配置了 setuptools `src` layout，`uv.lock` 锁定依赖。

**Subagent 策略**：一份详细的模型选择策略文档（`docs/subagent-strategy.md`），定义了按任务风险等级的模型选择基线表（从 `gpt-5.4-mini medium` 到 `gpt-5.5 high`）、升级/降级条件、Review 分级（低风险机械修改 → 主线程自检 → 独立 reviewer → 阶段验收）。

**文档体系**：5 份设计文档 + 7 份实施计划 + 9 份交接记录。交接记录不是流水账——每个都记录了当前断点、已完成验证、subagent 使用情况、下一次精确执行顺序和禁止事项。


## 总结

回到你的问题。核心流程确实是 adapter → predict → evaluate 三步，但代码量绝大部分花在了：

1. **安全防护**——四层数据隔离、manifest 检查、secret 脱敏、路径逃逸防护
2. **恢复能力**——状态机、全量 manifest 比对、原子写入、崩溃恢复
3. **工程保障**——15+ 项配置校验、类型安全检查、conversation 级并发隔离
4. **深度适配**——不修改第三方源码的前提下注入参数、重试逻辑、模块安全导入
5. **防御性设计**——付费前零副作用、只读 preflight、静默忽略 vs 显式拒绝的语义区分
6. **可观测性**——三路日志、进度快照、结构化事件
7. **可扩展性**——声明式注册、能力校验、动态 CLI
8. **向后兼容**——旧 runner 保留降级、新旧不能混跑、artifact 可复用
9. **测试**——fake 系统、TDD、markers 隔离

如果只做 happy path 的 "adapter → predict → evaluate"，代码量大概在 1,500-2,000 行。但我们做的是一个**真正可复现、可审计的评测框架**，这需要上面提到的所有这些东西。这就是为什么总共写了约 22,000 行代码。
