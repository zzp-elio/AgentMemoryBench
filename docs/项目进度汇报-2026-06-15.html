Section 03

基本功能之外 — 为什么代码量这么大？

核心流程只有 adapter → predict → evaluate 三步。如果只写 happy path，约 1,500 行就够了。实际写了 ~12,000 行源码 + ~10,000 行测试。下面解释多出来的每一块在防什么。

01 / 数据安全
四层私有数据防护
做了什么

递归扫描所有传入 method 的 dict/list，case‑insensitive 匹配 12 个私有键名（answer、gold_answer、evidence、judge_label 等），发现匹配即抛 DataLeakageError 并给出完整 JSONPath 定位。

def validate_no_private_keys(payload, path="$"): if isinstance(payload, dict): for key in payload: if key.lower() in PRIVATE_KEY_NAMES: raise DataLeakageError(f"{path}.{key}")
如果不做

conversation metadata 里不小心带了一个 "answer": "mental health" 字段 → gold answer 泄漏给 method → method "看到"了标准答案 → 整个 benchmark 的 F1 分数作废，论文评审直接拒。

后果：一次泄漏 = 全部实验结果不可信。这层检查是最后一道防线——前面还有数据模型层（to_public_dict 不序列化 gold_answers）、Runner 重建层（copy.deepcopy 清空 gold_answers={}）、Manifest 层（禁止 api_key/secret 等键名），四层独立安全网。
02 / 存储安全
原子写入 + fsync
做了什么

所有 JSON/JSONL 覆盖写入不是直接 open(p, 'w').write()，而是：写同目录临时文件 → flush() → os.fsync(fd) 强制刷盘 → os.replace() 原子替换。失败时 finally 清理临时文件。

with NamedTemporaryFile(dir=parent, delete=False) as tf: tf.write(content); tf.flush() os.fsync(tf.fileno()) os.replace(tmp_path, target_path)
如果不做

长实验跑了 2 小时，进程在 json.dump() 中途被 OOM killer 杀掉 → 磁盘上留下截断的 method_predictions.jsonl → resume 时 JSON 解析失败 → 2 小时白跑。

后果：目标文件永远不会处于"写了一半"的状态。最坏情况是残留一个临时文件（下次会被覆盖），原始文件始终完好。
03 / 存储安全
JSONL 崩溃恢复（Torn Tail）
做了什么

read_jsonl(recover_torn_tail=True) 读取时检测文件末尾：如果最后一行 JSON 损坏 且 没有换行终止符 → 判定为崩溃残留半行，静默丢弃。中间行的损坏或已带换行符的尾行损坏 → 仍报错（不是崩溃是数据本身坏了）。

if recover_torn_tail \ and line_index == last_non_empty \ and not line.endswith(("\n","\r")): break # 丢弃 torn tail
如果不做

append‑only 的 events.jsonl 在进程崩溃时，最后一行写了一半（{"timestamp":"2026-）→ resume 时 json.loads() 抛异常 → 整个 events 文件读不了 → resume 失败。

后果：区分"崩溃导致的坏数据"和"数据本身就有问题"。前者自动修复（丢弃最后半行继续跑），后者必须报错让人查。
04 / Resume
Resume 全量 Manifest 逐字段比对
做了什么

Resume 时不是"粗略看看 run_id 一样就行"，而是把 dataset_sha256、完整 policy（max_workers、conversation_ids、question_limit）、完整 method 配置、source fingerprint 全部拿出来，与磁盘上已有 manifest 做 == 逐字段比对。任何不匹配直接报错拒绝。

if existing != manifest: raise ConfigurationError( "Resume manifest mismatch: " "dataset, method or policy changed")
如果不做

先跑了 2 个 worker 的实验崩了 → 你改成 4 个 worker 后 resume → 前半段是 2 worker 的结果，后半段是 4 worker 的结果 → 论文里写"我们用 4 workers 跑的实验" → 评审发现数据指纹对不上 → 撤稿。

后果：Resume 不等于"随便改参数继续跑"。manifest 是实验的 DNA——变了就是另一个实验。不比对 = 实验结果不可复现。
05 / Resume
逐 Turn 状态机 in_flight 绝不自动 Resume
做了什么

逐 turn checkpoint 三态：ready → in_flight → completed。状态转换严格校验。关键决策：in_flight 状态发现后 拒绝自动 resume，必须人工介入。

if checkpoint.status == "in_flight": raise ConfigurationError( "Cannot auto-resume in_flight turn")
如果不做

API 调用发出去了 → 网络超时 → 客户端以为失败了，标记 in_flight → 但服务端其实处理成功了 → 自动 resume → 同一条 turn 被写入两次 → memory 里出现重复数据 → 检索结果污染。

后果：超时 ≠ 失败。in_flight 是不可判定状态——你不知道服务端到底处理了没有。自动 resume = 赌服务端失败了，赌错了数据就脏了。
06 / Method 适配
MemoryOS 安全模块导入
做了什么

官方 eval/utils.py 在 import 阶段就执行 OpenAI(api_key="")，新版 SDK 直接报错。Adapter 临时 monkeypatch openai.OpenAI 构造器（空 key 时注入占位值）→ 导入 7 个 eval 模块 → finally 恢复真实 OpenAI 类、清除 sys.modules 中已导入的模块名、移除临时 sys.path。

openai.OpenAI = safe_constructor try: import utils, short_term_memory, ... finally: openai.OpenAI = real_class sys.modules.pop("utils", None) sys.path.remove(eval_dir)
如果不做

import memoryos_adapter → 官方 eval/utils.py 执行 OpenAI(api_key="") → SDK 报 The api_key client option must be set → MemoryOS 整个跑不起来 → 这个 method 无法评测。

后果：研究脚本不是库——它在 import 时做网络初始化。不改第三方源码的前提下，monkeypatch 是唯一能让它跑起来的方式。
07 / 安全
成本保护前置（零副作用 Preflight）
做了什么

_confirm_prediction_cost() 在 读取 profile、dataset、.env、创建目录、构造 method 之前就执行。需要 API 但没传 --confirm-api → 直接报错退出。确认通过后才开始读配置、建目录。

def _confirm_prediction_cost(...): if requires_api and not confirm_api: raise ConfigurationError(...) # ↑ 这行以上：零副作用 # 这行以下才开始读 .env、建目录
如果不做

用户忘记传 --confirm-api → 程序先读了 .env 和 dataset → 然后才报错 → API key 已经加载到内存 → 后续的 error log 或 traceback 可能意外打印 key → secret 泄漏。

后果：Review 时发现初版确实有这个 bug——建了目录才报错，导致非法输入留下垃圾目录。修复后 preflight 在一切副作用之前执行。
08 / 配置校验
配置布尔陷阱防御
做了什么

Python 中 bool 是 int 的子类：isinstance(True, int) == True。如果用 isinstance(x, int) 校验，True 会通过。所以我们的校验函数用 type(x) is not int 严格拒绝 bool。

def _require_positive_int(value, field): if type(value) is not int or value <= 0: raise ConfigurationError(...) # type(True) is bool, not int → 被拒绝 ✓
如果不做

TOML 里误写 short_term_capacity = true → Python 解析为 True → isinstance(True, int) 返回 True → 校验通过 → 容量从 7 变成 1 → 每 1 条 turn 就触发一次 mid‑term 更新 → token 消耗暴增 7 倍 → $3 的实验变成 $21。

后果：花了 $21 跑出一个错误实验，因为 TOML 里的一个 true 被 Python 静默吞掉了。这个 bug 在 Python 社区非常常见。
09 / 安全
路径逃逸防护
做了什么

外部输入（conversation_id）不直接用于文件名，而是做 hashlib.sha256(id.encode()).hexdigest()，用 64 位十六进制字符串作为 checkpoint 文件名。同时 resolve_third_party_method_path() 校验路径不逃逸出 method 根目录。

digest = hashlib.sha256( conversation_id.encode("utf-8") ).hexdigest() path = root / f"{digest}.json"
如果不做

benchmark 数据里出现 conversation_id = "../../etc/passwd" → checkpoint 写到系统目录 → CI 环境里直接安全漏洞。或者 id 包含 / → 路径解析错误 → checkpoint 写到错误位置 → resume 找不到状态。

后果：外部数据不可信。直接拼接文件路径 = 路径注入漏洞。SHA-256 后永远只产生 [0-9a-f] 字符，绝对安全。
10 / 并发安全
并发 Worker 隔离 + 协调层串行提交
做了什么

Worker 线程只调 method（add + get_answer），返回不可变 _ConversationAnswerBatch（tuple）。协调线程用 as_completed() 收集结果，串行原子覆写共享 JSONL。Worker 之间不共享文件句柄。

futures = {executor.submit(worker, ...): cid for cid in pending} for future in as_completed(futures): batch = future.result() # 不可变 tuple atomic_write_jsonl(path, all_records)
如果不做

两个 worker 同时 json.dump() 追加到同一个 method_predictions.jsonl → 线程 A 写了一行的一半，线程 B 插进来写了另一行 → 文件里出现 {"question_id":"q1"...{"question_id":"q2"... → JSON 行交错 → 文件损坏 → 5 小时实验产物无法解析。

后果：线程并发写同一个文件 = 数据竞争 = 不可预测的文件损坏。协调层串行提交保证了正确性，只牺牲了极少的吞吐（写文件远快于 LLM 调用）。
