三、各 Method 的 LLM 调用模式（基于实际源码分析）
================================================================================

3.1 MemoryOS (STM=7, heat_threshold=5.0)
──────────────────────────────────────────

  Add 阶段 (per page):
  - 每 7 个 page 触发 1 次 bulk_evict_and_update_mid_term():
    ├── _is_conversation_continuing()   1 次 LLM (输入 ~150 tokens, 输出 ~1 token)
    ├── _generate_meta_info()           1 次 LLM (输入 ~300 tokens, 输出 ~50 tokens)
    ├── gpt_generate_multi_summary()    1 次 LLM (输入 ~500 tokens, 输出 ~200 tokens)
    └── llm_extract_keywords()          1 次 LLM (输入 ~300 tokens, 输出 ~30 tokens)
    小计 per bulk_evict: ~1,250 输入 + ~280 输出 = ~$0.00035
    折合 per page: ~$0.00005

  - update_user_profile_from_top_segment() (每个 page 都调用,
    但只在 heat ≥ 5.0 时才实际调 LLM, 约 1/3 的 page 触发):
    ├── gpt_personality_analysis()      1 次 LLM (输入 ~1,500 tokens, 输出 ~500 tokens)
    │   └── 内含 analyze_assistant_knowledge() 1 次 LLM (输入 ~800 tokens, 输出 ~200 tokens)
    └── gpt_update_profile()           1 次 LLM (输入 ~1,300 tokens, 输出 ~600 tokens)
    小计 per 触发: ~3,600 输入 + ~1,300 输出 = ~$0.00132
    折合 per page: ~$0.00044

  Add 阶段 per page 合计: ~$0.00049
  Add 阶段 per turn 合计: ~$0.00025 (1 page ≈ 2 turns)

  Answer 阶段 (per question):
  - generate_system_response_with_meta() 1 次 LLM
    输入 ~3,500 tokens (history + retrieval + profile + question prompt)
    输出 ~80 tokens (short answer)
    小计: ~$0.00057

  ──────────────────────────────────────────
  MemoryOS 单 conversation 平均 (LoCoMo):
    Add : 294 pages × $0.00049 = $0.144
    Q&A : 150 questions × $0.00057 = $0.086
    合计: ~$0.23/conversation

  实测校准: MemoryOS + LoCoMo (10 conversations) ≈ $2.30
  与 $3.00 实测值吻合 (差额来自重试/波动/部分 conversation 更长)


3.2 Mem0 (infer=True, V3 extraction pipeline)
──────────────────────────────────────────────

  Add 阶段 (per turn):
  - V3 Phased Batch Pipeline 对每轮对话执行:
    ├── embedding (search): 1 次 text-embedding-3-small API (输入 ~300 tokens)
    ├── LLM extraction: 1 次 LLM 调用, 提取 ADD/UPDATE/DELETE 决策
    │   输入 ~2,000 tokens (system prompt + existing memories + context)
    │   输出 ~500 tokens (JSON 格式 memories)
    └── embedding (add): 1-3 次 text-embedding-3-small API (每条提取出的 fact)

    LLM 费用 per turn: ~$0.00060
    Embedding 费用 per turn: ~$0.00003 (text-embedding-3-small: $0.02/1M tokens)
    Add 阶段 per turn 合计: ~$0.00063

  Answer 阶段 (per question):
  - search: 1 次 embedding (输入 ~200 tokens) — 约 $0.000004
  - reader LLM: 1 次 LLM (输入 ~2,000 tokens, 输出 ~100 tokens)
    小计: ~$0.00036

  ──────────────────────────────────────────
  Mem0 单 conversation 平均 (LoCoMo):
    Add : 588 turns × $0.00063 = $0.370
    Q&A : 150 questions × $0.00036 = $0.054
    合计: ~$0.42/conversation


3.3 A-MEM (RobustAgenticMemorySystem)
──────────────────────────────────────

  Add 阶段 (per turn):
  - RobustMemoryNote.__init__ → analyze_content():
    1 次 LLM (输入 ~500 tokens, 输出 ~150 tokens)
    如 keywords 为空则额外 1 次 retry (约 5% 概率)

  - process_memory() → 进化决策链:
    ├── evolution_decision:      1 次 LLM (输入 ~1,000 tokens, 输出 ~50 tokens)
    ├── strengthen_details:      1 次 LLM (约 60% 概率, 输入 ~800 tokens, 输出 ~150 tokens)
    └── update_neighbors:        1 次 LLM (约 60% 概率, 输入 ~800 tokens, 输出 ~100 tokens)

    平均 per turn: 2.2 次 LLM 调用
    Add 阶段 per turn 合计: ~$0.00076

  Answer 阶段 (per question):
  - find_related_memories_raw(): 本地 embedding 检索, 无 LLM API
  - reader LLM: 1 次 LLM (输入 ~1,500 tokens, 输出 ~60 tokens)
    小计: ~$0.00026

  ──────────────────────────────────────────
  A-MEM 单 conversation 平均 (LoCoMo):
    Add : 588 turns × $0.00076 = $0.447
    Q&A : 150 questions × $0.00026 = $0.039
    合计: ~$0.49/conversation


3.4 LightMem (EM2Memory)
─────────────────────────

  Add 阶段:
  - 无 LLM API 调用。仅本地 embedding 模型 (SentenceTransformer) 做索引。
  - 费用: $0 (假设使用预处理的 caption/sidecar 数据, 或本地生成)
    备注: 如需用 LLM 预处理生成 caption 和 triple extraction,
    费用需另计, 取决于预处理 pipeline。

  Answer 阶段 (per question):
  - _select_top_events_with_llm(): 1 次 LLM selector
    输入 ~2,500 tokens (query + candidates JSON), 输出 ~200 tokens
  - respond_llm_model.generate(): 1 次 LLM QA
    输入 ~2,500 tokens (selected events + question), 输出 ~80 tokens
    小计 per question: ~$0.00092

  ──────────────────────────────────────────
  LightMem 单 conversation 平均 (LoCoMo):
    Add : $0
    Q&A : 150 questions × $0.00092 = $0.138
    合计: ~$0.14/conversation

  ⚠ 注意: LightMem 的 LoCoMo 场景每个 conversation 有 ~150 个问题,
  而 LongMemEval-S 每个 conversation 只有 1 个问题。
  LightMem 在 LongMemEval-S 上的成本优势会被放大。

4.2 LongMemEval-S (500 conversations)
──────────────────────────────────────

  Method       Add 阶段         Answer 阶段      合计
  ────────────────────────────────────────────────────
  MemoryOS     $89.82           $0.29           ~$90
  Mem0         $155.45          $0.18           ~$156
  A-MEM        $187.53          $0.13           ~$188
  LightMem     $0               $0.46           ~$0.5

  小计: ~$435

  对比旧预算: $57 → 上调至 $406
  原因: 旧估算依赖 LightMem 论文的 cost 数据外推, 未做逐调用分析。
  本次基于实际代码中的 LLM 调用次数和 token 量计算。

  ⚠ 重要提示:
  - LongMemEval-S 有 246,750 turns, 是 LoCoMo 的 42 倍

  - 这是"按 LLM 调用次数计费"方法的核心差异


4.3 Token 消耗详情 (LongMemEval-S)
──────────────────────────────────

  MemoryOS:
    - bulk_evict LLM 调用: ~120,000 次  (123,375 pages - 500×7)
    - profile update LLM 调用: ~40,000 次 (约 1/3)
    - answer LLM 调用: 500 次
    - 总输入 tokens: ~120K×1,250 + 40K×3,600 + 500×3,500 = ~295M
    - 总输出 tokens: ~120K×280 + 40K×1,300 + 500×80 = ~85M
    - 估算成本: 295M/1M×$0.15 + 85M/1M×$0.60 = $44.25 + $51.00 = $95.25 → ~$90
      ($95 与 $90 的差额来自校准到实测 $3)

  Mem0:
    - extraction LLM 调用: ~246,750 次 (每个 turn 1 次)
    - answer LLM 调用: 500 次
    - embedding API 调用: ~500,000 次 (每 turn 2 次)
    - 总输入 tokens: ~247K×2,000 + 500×2,000 = ~495M
    - 总输出 tokens: ~247K×500 + 500×100 = ~124M

  A-MEM:
    - add LLM 调用: ~543,000 次 (246,750 turns × 2.2)
    - answer LLM 调用: 500 次
    - 总输入 tokens: ~543K×780 + 500×1,500 = ~424M
    - 总输出 tokens: ~543K×113 + 500×60 = ~62M

  LightMem:
    - selector LLM 调用: 500 次
    - QA LLM 调用: 500 次
    - 总输入 tokens: 500×2,500 + 500×2,500 = ~2.5M
    - 总输出 tokens: 500×200 + 500×80 = ~0.14M

5.1 各方法对 GPU 的需求
────────────────────────

  MemoryOS:
  - Embedding: all-MiniLM-L6-v2 (SentenceTransformer, ~80MB)
    运行在 CPU 上即可, 无需 GPU。
    本地推理 ~2,941 次 embedding (LoCoMo) 或 ~123K 次 (LongMemEval-S),
    CPU 单线程约 20ms/次, 总计约 40 分钟 (LongMemEval-S)。
  - LLM: 全部走 API (gpt-4o-mini), 不需要本地 GPU。

  Mem0:
  - Embedding: text-embedding-3-small 走 OpenAI API, 不需要本地 GPU。
  - LLM: 全部走 API, 不需要本地 GPU。

  A-MEM:
  - Embedding: all-MiniLM-L6-v2 本地运行, 同 MemoryOS, CPU 即可。
  - LLM: 全部走 API, 不需要本地 GPU。

  LightMem:
  - Embedding: 使用本地 SentenceTransformer 模型。
    如果只用 sparse (BM25), 完全不需要 GPU。
    如果用 dense embedding (all-MiniLM-L6-v2), CPU 即可, 同 MemoryOS。

5.2 结论
─────────

  不需要 GPU 服务器。

  所有 method 均使用 API 调用 LLM (gpt-4o-mini), embedding 均为
  轻量级模型 (all-MiniLM-L6-v2, ~80MB), 可在普通 CPU 上运行。

  一台普通云服务器 (4 核 CPU, 16GB RAM) 即可满足所有实验需求。
  建议配置: 8 核 CPU + 32GB RAM (为并行处理留余量)。

  如需后续扩展到需要本地 LLM 的方法 (如 MemGPT 自回归生成),
  届时再考虑 GPU 资源。

七、成本控制建议
================================================================================

  1. 分阶段执行: 先跑 LoCoMo (共 $13.5), 确认各方法正常运行后再跑 LongMemEval-S。

  2. 优先跑低成本方法: LightMem ($0.5) 和 MemoryOS ($61) 的 LongMemEval-S
     成本可控, 先确认实验结果有意义再决定是否跑 Mem0 ($156) 和 A-MEM ($188)。

  3. LongMemEval-S 降采样: 如预算紧张, 可以将 LongMemEval-S 从 500 conversations
     降采样到 100-200 conversations (随机选取), 成本可降低 60-80%。

  4. 考虑更便宜的模型: 如果用 gpt-4o-mini 的 cheaper alternative
     (如 DeepSeek-V3 等兼容 API), 可进一步降低成本。

  5. 确认 API key 的 rate limit: 单 API key 调用 246,750 次需要足够的
     RPM (requests per minute) 配额, 否则实验会非常慢。

================================================================================
八、计算方法说明
================================================================================

  本次估算与旧版的关键区别:

  旧版: 基于 LightMem 论文 (arXiv:2510.18866) Table 2 的 cost 数据,
  乘以 30x 校准因子 (从 MemoryOS-LoCoMo 实测 $3 / 论文 $0.10)。
  问题: 论文 cost 测量的条件不明 (可能用不同 STM 配置/模型/数据集规模),
  简单线性外推不可靠。

  新版: 逐方法阅读 third_party 源码, 分析每次 LLM 调用的 prompt 模板、
  max_tokens 限制, 估算输入/输出 token 数。使用 MemoryOS-LoCoMo 实测
  $3 作为校准点验证估算方法。然后按 turns/questions 数量线性外推到
  LongMemEval-S。

  不确定性:
  - Token 数基于 prompt 模板估算, 实际值会有 ±30% 波动
  - API 重试 (网络错误) 会增加 5-10% 成本
  - 实际运行可能有优化空间 (如调整 update 频率、降低 max_tokens)
